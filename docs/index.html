Current capabilities

What the v1 alpha SDK can return today.

The first release is intentionally focused: it proves the shared SDK contract, native package model, validation behavior, and OS-backed randomness before expanding into richer random data.

int

Random integers

Generate whole numbers with a default range of 0..99 or pass explicit inclusive bounds such as randomInt(10, 20).

float

Random floats

Generate floating-point values with a default range of 0.0..1.0 or provide your own lower and upper bounds.

char

Random characters

Generate printable ASCII characters with a default range of "A".."Z" or choose a bounded character range.

Why developers use it

One contract, native packages.

The SDKs follow a shared v1 contract, but each package uses the language's own secure random facilities. Your users get an API that feels local, without a service call or CLI bridge in the happy path.

Language agnostic by design

The API should feel native everywhere.

libRandomizer is not trying to make every language call a Python script. The repo defines one shared behavior contract, then gives each language its own package and idiomatic import path.

Shared contract

Defaults, inclusive ranges, printable ASCII rules, equal-boundary behavior, and clear failures are documented once in spec/v1.

Native implementations

Each SDK uses that platform's secure random facilities instead of depending on a hosted service or language bridge for normal use.

Consistent developer experience

The goal is a tiny call surface: import the package, call randomInt(), randomFloat(), or randomChar(), and keep moving.

What alpha v1 includes

Small enough to trust, useful enough to ship.

v1 alpha is the foundation for a much larger randomization SDK. It already covers native packages, the Python reference API, a JSON CLI fallback, shared validation rules, and the MIT license.

  • Default integer range: 0..99
  • Default float range: 0.0..1.0
  • Default character range: "A".."Z"
  • Bounded calls for all v1 types
  • Printable ASCII validation for characters
  • JSON CLI fallback for bridge scenarios

Long-term goal

A language-agnostic SDK for random data of almost any kind.

The north star is simple but ambitious: no matter what language a developer is using, they should be able to ask libRandomizer for a random value that matches the type they need.

Today that means ints, floats, and chars. Future versions should add common random domains like strings, symbols, colors, words, names, places, dates, choices, weighted values, and structured random objects while preserving the same low-friction API style.
